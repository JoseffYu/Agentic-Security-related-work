# Related Literature Table -- Verified Staged Authorization

This table summarizes the core literature for the Verified Staged Authorization project. The "abstract main point" column paraphrases what the paper's abstract is about; it is not a verbatim abstract.

Status legend:

- **Peer-reviewed**: published in a conference, journal, workshop, or proceedings.
- **Preprint**: arXiv or similar artifact; useful, but venue status should be rechecked before camera-ready.
- **Technical report / guidance**: non-venue source used for background or framing.

| 1 | Prompt injection threat | [Greshake et al., "Not What You've Signed Up For: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection"](https://arxiv.org/abs/2302.12173) | AISec 2023 workshop / arXiv | External content retrieved by an LLM application can contain instructions that override the developer's intended behavior. | 2 | Prompt injection threat | [Liu et al., "HouYi: A Prompt Injection Attack Framework for Large Language Models"](https://arxiv.org/abs/2306.05499) | USENIX Security 2024 | Presents a black-box framework for discovering prompt injection attacks against LLM-integrated applications and evaluates it on real systems. **Indirected injection attack**| Shows prompt injection is systematic and practical, not an isolated toy problem. |
| 3 | Prompt injection formalization | [Liu et al., "Formalizing and Benchmarking Prompt Injection Attacks and Defenses"](https://arxiv.org/abs/2310.12815) | USENIX Security 2024 | Defines prompt injection attacks and defenses more formally and introduces benchmark methodology for measuring attack and defense behavior. **Formalise prompt injection attack**| Gives the threat area formal vocabulary, but VSA extends the target from one prompt interaction to staged agent execution. |

| 9 | RAG poisoning | [Zou et al., "PoisonedRAG: Knowledge Poisoning Attacks to Retrieval-Augmented Generation of Large Language Models"](https://arxiv.org/abs/2402.07867) | USENIX Security 2025 / arXiv | Demonstrates that injecting a small number of adversarial documents into a retrieval corpus can compromise RAG answers for target queries. **Not necessary**| Shows retrieved knowledge must enter as low-integrity observation, not as trusted fact. |
| 10 | Confused deputy in RAG | [Anantha-Padmanaban et al., "ConfusedPilot: Confused Deputy Risks in RAG-based LLMs"](https://arxiv.org/abs/2408.04870) | Preprint | Argues that RAG assistants can behave as confused deputies when information gathered under one authority context is used under another. **Not necessary**| Gives the closest classical-security framing for authority laundering. |
| 11 | Action hijacking | [Zhang et al., "Towards Action Hijacking of LLM-based Agents"](https://arxiv.org/abs/2412.10807) | Preprint | Studies attacks that manipulate an agent's action sequence rather than only its text output, causing harmful action assembly through benign-looking inputs. **Can act as threat model**| Closest threat analogue to VSA's observation-to-authorized-action laundering path. |

| 13 | Structured prompt defense | [Chen et al., "StruQ: Defending Against Prompt Injection with Structured Queries"](https://www.usenix.org/conference/usenixsecurity25/presentation/chen-sizhe) | USENIX Security 2025 | Uses structured prompt/query formats and model training so the model learns to follow only the instruction channel, not the data channel. **phrasing prompt/mark prompt (see how they do this phrasing)**| Addresses instruction/data separation at the model interface; VSA enforces separation at runtime stages. |
| 14 | Alignment defense | [Chen et al., "SecAlign: Defending Against Prompt Injection with Preference Optimization"](https://arxiv.org/abs/2410.05451) | ACM CCS 2025 / arXiv | Uses preference optimization on secure/insecure response pairs to make models less likely to obey injected instructions. **phrasing prompt/mark prompt (see how they do this phrasing), but this one based on preference. still untrusted**| Model-level mitigation; VSA remains needed when the model still proposes unsafe actions. |
| 15 | Guard defense | [Xiang et al., "GuardAgent: Safeguard LLM Agents by a Guard Agent via Knowledge-Enabled Reasoning"](https://arxiv.org/abs/2406.09187) | Preprint | Adds a guard agent that reasons over safety knowledge and blocks unsafe agent behavior. **LLM-as-judge**| Contrasts with VSA's design choice: LLMs can propose, but a deterministic checker authorizes. |
| 16 | Agent runtime defense | [Debenedetti et al., "CaMeL: Defeating Prompt Injections by Design"](https://arxiv.org/abs/2503.18813) | Preprint | Redesigns agent execution around a control/data separation model, capabilities, and a custom interpreter to prevent untrusted data from influencing privileged behavior. **Taming Various Privilege said this is dual-llm framework**| Closest competitor; VSA differs by tracking staged derivation events and authority promotion paths, not only values in an interpreter. |
| 17 | Agent IFC defense | [Wu et al., "A System-Level Defense against Indirect Prompt Injection Attacks: An Information Flow Control Perspective"](https://arxiv.org/abs/2409.19091) | Preprint | Applies IFC ideas to LLM systems by separating components, producing structured plans, and monitoring how untrusted inputs affect planning.**Relevant work, worth to read it** | Strongly adjacent; VSA adds authority algebra, witness-backed promotion, and staged preservation. |

## Shortlist of Closest Competitors

For the actual paper, the closest direct comparisons should be:

| Work | Closest overlap | One-line difference from VSA |
|---|---|---|
| SEAgent | Privilege escalation defense with MAC/ABAC and information-flow graph | SEAgent is entity-attribute/access-control oriented; VSA is action-level provenance plus authority derivation and pre-execution authorization. |
| PFI | Privilege escalation prevention via prompt-flow integrity | PFI validates unsafe prompt/data flows; VSA verifies RDG-backed action authorization and promotion witnesses. |
| CaMeL | Secure-by-design LLM-agent runtime | CaMeL protects generated programs and values; VSA protects staged derivation and authorization transitions. |
| f-secure | IFC for indirect prompt injection | f-secure controls untrusted information flow into planning; VSA also checks authority introduction and promotion witnesses. |
| Progent | Programmable tool privilege control | Progent controls tool permissions; VSA controls permissions as a function of provenance closure. |
| AgentSpec | Runtime policy enforcement | AgentSpec monitors runtime behavior; VSA adds an event-centered RDG and staged preservation property. |
| Agent-C | Formal temporal constraints for tool calls | Agent-C checks temporal properties; VSA checks authorization, integrity, provenance, and authority promotion. |
| ShieldAgent | Verifiable safety policy reasoning over action trajectories | ShieldAgent reasons with a guard agent and policy circuits; VSA uses deterministic runtime authorization as the security boundary. |
| IsolateGPT | Execution isolation | IsolateGPT isolates execution contexts; VSA mediates within-agent stage transitions. |
| ToolHijacker / ObliInjection | Agent-specific prompt injection against tool selection and multi-source inputs | These are threat papers; VSA supplies a runtime authorization architecture intended to stop the resulting unsafe tool introduction or action promotion. |
| StruQ / SecAlign / Spotlighting | Prompt/model-level defenses | These reduce model obedience to injected instructions; VSA verifies authorization outside the model. |
