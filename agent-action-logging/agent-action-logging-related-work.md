# Related Literature Table -- Agent Action Logging and Provenance

This table summarizes literature related to automatically recording, tracing, auditing, or reconstructing LLM-agent actions. The "What the abstract is saying" column paraphrases the paper's abstract-level contribution; it is not a verbatim abstract.

Status legend:

- **Peer-reviewed**: published in a conference, journal, workshop, or proceedings.
- **Preprint**: arXiv or similar artifact; useful, but venue status should be rechecked before camera-ready.
- **Technical report / guidance**: non-venue source used for background or framing.

| # | Category | Work | Status | What the abstract is saying | Why it matters for agent action logging | comment |
|---:|---|---|---|---|---|---|
| 1 | Agent observability taxonomy | [Dong et al., "AgentOps: Enabling Observability of LLM Agents"](https://arxiv.org/abs/2411.05285) | Preprint, arXiv 2024 | This work surveys AgentOps tooling and proposes a taxonomy of artifacts that should be traced across an agent lifecycle, including goals, plans, tool calls, intermediate outputs, and dependencies among them. | Good framing paper: it explains what should be logged before designing a concrete logger. It is more taxonomy/reference architecture than enforcement mechanism. | It treats logging as lifecycle observability: collect design-time and runtime artifacts, connect them into traces, and use those traces for monitoring, anomaly detection, and accountability. |
| 2 | Structured runtime logging | [AlSayyad et al., "AgentTrace: A Structured Logging Framework for Agent System Observability"](https://arxiv.org/abs/2602.10133) | Preprint, arXiv 2026 | AgentTrace proposes schema-based runtime instrumentation for LLM agents, capturing operational, cognitive, and contextual traces for security, accountability, and failure analysis. | Closest match to "automatically log agent actions as they happen." It is explicitly about structured logging for agent observability. | It records nested spans for method calls, prompts, reasoning/reflection steps, tool invocations, timing, and external I/O. The paper positions OpenTelemetry-style telemetry as the carrier, so agent events become structured trace records rather than raw text logs. |
| 3 | System-level tracing | [Zheng et al., "AgentSight: System-Level Observability for AI Agents Using eBPF"](https://arxiv.org/abs/2508.02736) | Preprint / PACMI@SOSP listing, 2025 | AgentSight proposes boundary tracing for AI agents: observe both high-level LLM intent and low-level system effects from outside the agent code, then correlate them. | Relevant if the project wants automatic logs without modifying every agent framework. It is stronger than application-level logging because it observes OS/network boundaries. | It uses eBPF/kernel-level monitoring for system calls and process effects, plus boundary observation of LLM traffic, then correlates intent with effects. In practice, this is a "black-box-ish" action recorder: file writes, process launches, network calls, and LLM requests can be linked into one trace. |
| 4 | Structural testing traces | [Kohl et al., "Automated structural testing of LLM-based agents: methods, framework, and case studies"](https://arxiv.org/abs/2601.18827) | Preprint, arXiv 2026 | This paper adapts software structural testing to LLM agents by using OpenTelemetry traces, mocking, and automated assertions to expose internal agent workflows. | Useful for test-time action logging and regression testing. It shows how logs can be generated in a standard telemetry format and then used for assertions. | It captures agent trajectories as OpenTelemetry traces, including component interactions and tool calls; mocks LLM outputs to make traces reproducible; then checks trace-level assertions instead of only checking final answers. |
| 5 | Event-sourced agent runtime | [Nakajima, "The Log is the Agent: Event-Sourced Reactive Graphs for Auditable, Forkable Agentic Systems"](https://arxiv.org/abs/2605.21997) | Preprint, arXiv 2026 | ActiveGraph inverts the usual agent architecture: the append-only event log is the source of truth, and the working graph is a deterministic projection of that log. | Very relevant architecturally: it does not bolt logging onto an agent; it makes every agent state transition an event first. | Each model call, graph mutation, artifact creation, and reactive behavior emits an append-only event. Replay, fork, and lineage are obtained by re-projecting graph state from the log, so action history is complete by construction. |
| 6 | Runtime reference monitor / provenance | [Chinaei, "Causality Laundering: Denial-Feedback Leakage in Tool-Calling LLM Agents"](https://arxiv.org/abs/2604.04035) | Preprint, arXiv 2026 | The paper identifies causality laundering, where denied tool calls leak information through later benign actions, and proposes an Agentic Reference Monitor with denial-aware provenance. | Important for security logs: it shows that logs must include denied actions, not only successful actions. | ARM mediates every tool invocation and records tool calls, returned data, field-level provenance, and denied actions in a provenance graph. It adds counterfactual edges from denied actions so later outputs can be checked for denial-induced influence. |
| 7 | Offline provenance audit | [Cai et al., "Ghost in the Agent: Redefining Information Flow Tracking for LLM Agents"](https://arxiv.org/abs/2604.23374) | Preprint, arXiv 2026 | NeuroTaint audits LLM-agent execution traces offline and reconstructs source-to-sink information flow using a Dynamic Context Provenance Graph, semantic similarity, causal analysis, and persistent context tracking. | Useful if the project records raw trajectories first and performs security analysis later. | It assumes traces already exist, then builds a provenance graph over context, tool outputs, memory, decisions, and sinks. The key logging implication is that trace records need enough detail to recover semantic influence, not just exact string copies. |
| 8 | Argument-level provenance | [Fan et al., "The Granularity Mismatch in Agent Security: Argument-Level Provenance Solves Enforcement and Isolates the LLM Reasoning Bottleneck"](https://arxiv.org/abs/2605.11039) | Preprint, arXiv 2026 | PACT argues that monitoring whole tool calls is too coarse; security should track provenance at the level of individual tool arguments and semantic roles. | Directly relevant for permission logs: an action log should record which argument carried authority, target, command, credential, or payload. | It assigns semantic roles to tool arguments and tracks value provenance across replanning steps. Instead of logging only `tool=send_email`, it distinguishes whether `recipient`, `body`, or `credential` came from trusted user input, untrusted web data, memory, or model generation. |
| 9 | Decision provenance | [Wang et al., "MindGuard: Tracking, Detecting, and Attributing MCP Tool Poisoning Attack via Decision Dependence Graph"](https://arxiv.org/abs/2508.20412) | Preprint, arXiv 2025 | MindGuard models tool invocation decisions with a Decision Dependence Graph derived from attention-like signals, enabling detection and attribution of tool poisoning attacks. | Useful when the target is not just "what action happened" but "why this tool was selected." | It logs or reconstructs decision-level dependencies between tool metadata, reasoning concepts, and tool invocation. This helps attribute a suspicious action to poisoned descriptions even when the poisoned tool was not actually executed. |
| 10 | Influence provenance for prompt injection | [Weng et al., "ARGUS: Defending LLM Agents Against Context-Aware Prompt Injection"](https://arxiv.org/abs/2605.03378) | Preprint, arXiv 2026 | ARGUS constructs an influence provenance graph to track how untrusted context propagates into agent decisions under context-aware prompt injection. | Relevant for log design because it says action logs need edges from context sources to decisions, not just a flat chronological trace. | It records which dynamic context pieces influenced a decision and verifies whether the decision is justified by trusted evidence before execution. This is closer to provenance-aware auditing than simple event logging. |
| 11 | Privacy disclosure logging | [Stanley et al., "An AI Agent Execution Environment to Safeguard User Data"](https://arxiv.org/abs/2604.19657) | Preprint, arXiv 2026 | GAAP proposes an execution environment that enforces user-specified privacy permissions and tracks disclosure of private data across tasks. | Important for agents handling private data: logs must persist across sessions, because data learned in one task can leak in a later task. | It isolates private data in controlled stores, annotates tool access and disclosure events, and maintains a persistent disclosure log so later actions can be checked against prior flows and user permissions. |
| 12 | Checkpoint/restore action log | [Zheng et al., "ACRFence: Preventing Semantic Rollback Attacks in Agent Checkpoint-Restore"](https://arxiv.org/abs/2603.20625) | Preprint, arXiv 2026 | ACRFence studies semantic rollback attacks in which restored agents regenerate semantically equivalent but syntactically different side-effecting tool calls. | Relevant if an agent system supports retries, branching, or checkpoint restore. A normal log of exact calls is insufficient. | It records irreversible tool effects and enforces replay-or-fork semantics after restore. The log must capture action intent and side-effect identity, not only the raw serialized request, so duplicate payments or credential reuse can be blocked. |
| 13 | Runtime security hooks and audit plane | [Li, "OpenClaw PRISM: A Zero-Fork, Defense-in-Depth Runtime Security Layer for Tool-Augmented LLM Agents"](https://arxiv.org/abs/2603.11853) | Preprint, arXiv 2026 | PRISM proposes a runtime layer for tool-augmented agents with lifecycle hooks, risk accumulation, policy controls, and a tamper-evident audit plane. | Useful as an implementation pattern for automatic action logging inside an agent gateway. | It intercepts message ingress, prompt construction, tool execution, tool-result persistence, outbound messaging, sub-agent spawning, and startup. Those hooks emit audit events and support hot-reloadable policies, so logs are tied to enforcement points. |
| 14 | Safe in-the-wild testing monitor | [Naihin et al., "Testing Language Model Agents Safely in the Wild"](https://arxiv.org/abs/2311.10538) | Preprint, arXiv 2023 | The paper proposes AgentMonitor, a context-sensitive monitor for safely testing language-model agents on the open internet and stopping unsafe behavior. | Early paper for auditing live agent actions in uncontrolled environments. | It wraps existing agents with a monitor that audits actions, ranks suspect behavior, logs cases for human review, and halts unsafe tests. It is less detailed than newer provenance systems but useful as a safety-monitoring baseline. |
| 15 | Temporal trace monitoring | [Sheffler, "An Approach to Checking Correctness for Agentic Systems"](https://arxiv.org/abs/2509.20364) | Preprint, arXiv 2025 | This work proposes a temporal expression language for checking agent behavior over sequences of tool calls and state transitions. | Useful when logs are intended to feed runtime verification formulas like "A must happen before B" or "tool X cannot follow event Y." | It records execution traces of tool invocations and state transitions, then evaluates temporal assertions over the trace. The logging method is event-sequence oriented rather than provenance-graph oriented. |
| 16 | Formal auditing and runtime monitoring | [Alamdari et al., "Formal Methods Meet LLMs: Auditing, Monitoring, and Intervention for Compliance of Advanced AI Systems"](https://arxiv.org/abs/2605.16198) | Preprint, arXiv 2026 | TRAC uses formal temporal constraints, offline auditing, online runtime monitoring, predictive monitoring, and intervention for LLM-based systems. | Useful bridge from "record actions" to "verify actions against temporal rules." | It labels runtime events, evaluates Linear Temporal Logic constraints over event histories, and can intervene before predicted violations. The key dependency is a reliable event stream of agent actions and states. |
| 17 | Cognitive observability | [Rombaut et al., "Watson: A Cognitive Observability Framework for the Reasoning of LLM-Powered Agents"](https://arxiv.org/abs/2411.03455) | Peer-reviewed, ASE 2025 | Watson introduces cognitive observability: retroactively infer implicit reasoning traces in fast-thinking agents without changing their behavior. | Relevant if action logs alone are too shallow and the project wants post-hoc explanations for why an action occurred. | It uses prompt attribution to infer reasoning traces after the fact. It is not a low-level action logger, but it can enrich action logs with inferred decision rationale. |
| 18 | Scientific workflow provenance | [Souza et al., "LLM Agents for Interactive Workflow Provenance: Reference Architecture and Evaluation Methodology"](https://arxiv.org/abs/2509.13978) | Peer-reviewed workshop listing, SC25-W 2025 | This paper uses LLM agents to query and analyze runtime workflow provenance across Edge-Cloud-HPC scientific workflows. | Useful background from workflow provenance: logs are structured metadata over who/what/when/how, then agents query them. | It assumes workflow systems already emit provenance metadata and builds a RAG/agent interface over that metadata. This is more "agent over logs" than "logging inside agents," but the provenance schema is informative. |
| 19 | Auditable domain agent platform | [Unlul et al., "An Auditable Agent Platform For Automated Molecular Optimisation"](https://arxiv.org/abs/2508.03444) | Preprint, arXiv 2025 | The paper builds a hierarchical multi-agent platform for molecular optimization where tool calls and molecule lineage are stored for auditability. | Domain-specific but useful: it shows how to make tool-heavy scientific agents auditable. | Each tool call is summarized and stored as provenance; molecular candidates maintain lineage across generation, editing, docking, and ranking. The log supports auditability and reuse of successful transformations. |
| 20 | Agent evaluation reproducibility | [Li and Storhaug, "Reproducible, Explainable, and Effective Evaluations of Agentic AI for Software Engineering"](https://arxiv.org/abs/2604.01437) | Preprint / position paper, arXiv 2026 | This position paper argues that software-engineering agent evaluations should publish Thought-Action-Result trajectories and LLM interaction data. | Useful for deciding what minimum trace data should be stored and released for reproducible agent evaluation. | It recommends recording TAR trajectories: thought, selected action, action result, and LLM interaction data. Even if private fields are redacted, the paper argues summarized trajectories are necessary for explainable comparison. |

## Shortlist of Closest Competitors

For an agent action logging project, the closest direct comparisons should be:

| Work | Closest overlap | One-line difference from agent action logging |
|---|---|---|
| AgentTrace | Schema-based runtime logging of agent behavior, tool calls, prompts, reasoning, external I/O, and timing. | Closest general-purpose logging framework; a new project should differ by stronger permission semantics, finer-grained provenance, or integration with runtime verification. |
| AgentSight | Framework-agnostic system-level tracing using eBPF and boundary observation. | Strong for black-box action capture, but it focuses on correlating intent and low-level system effects rather than defining permission-aware event semantics. |
| ActiveGraph | Event-sourced architecture where the log is the source of truth for replay, fork, and lineage. | Strong architectural competitor; differs if the new project logs existing agents without requiring an event-sourced runtime rewrite. |
| ARM / Causality Laundering | Runtime mediation of every tool call with provenance over successful and denied actions. | Closest for security-sensitive tool-call logging; narrower focus on laundering and denial-feedback leakage. |
| PACT | Argument-level provenance for authority-bearing tool arguments. | More precise than call-level logs; differs if the new project also logs stage transitions, user grants, and policy checks. |
| NeuroTaint | Offline provenance auditor over execution traces. | Strong post-hoc analysis baseline; differs from online logging if the new project emits structured events at execution time. |
| GAAP | Persistent disclosure log and data-flow tracking for private user data. | Strong privacy-focused baseline; narrower scope around confidentiality and user permission specifications. |
| PRISM | Lifecycle hooks and tamper-evident audit plane for an agent gateway. | Practical runtime security layer; differs if the new project is framework-independent or formally specified. |

## Notes

- The most common implementation pattern is **runtime instrumentation**: wrap agent framework calls, tool calls, model calls, memory reads/writes, and outbound effects, then emit structured events.
- A second pattern is **boundary tracing**: observe process, filesystem, network, and model-provider boundaries from outside the agent; AgentSight is the clearest example.
- A third pattern is **event sourcing**: make every state transition an append-only event, then derive agent state from the log; ActiveGraph is the clearest example.
- For security work, flat chronological logs are usually insufficient. The stronger papers add **provenance edges**: source data -> prompt/context -> decision -> tool argument -> external effect.
- Several papers imply that denied or blocked actions must be logged. ARM is the clearest example: a denied call can itself leak information through later behavior.
- For a permission-verification project, the log schema should likely include at least: user request, agent identity, stage, model call, tool call, tool arguments, argument provenance, permission grant/deny decision, policy ID, data labels, tool result, memory write/read, outbound effect, and final verdict.
- Venue status should be rechecked before citation. Much of this specific agent-observability literature is very recent and currently appears as arXiv/preprint material.
